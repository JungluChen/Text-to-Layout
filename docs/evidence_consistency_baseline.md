openems_result.s2p: 401/401 S-parameter samples are non-finite (NaN/Inf)

## C2 — CPW: the published value is not reproducible from its own solver output

`examples/showcase/02_cpw_50ohm/`. The Touchstone file is genuinely all-finite
(401 points) and the openEMS energy-decay convergence gate passes, so the run
is real. But re-extracting from the *committed* output gives a different number:

| Quantity | Committed | Recomputed from committed `.s2p` |
| --- | --- | --- |
| `characteristic_impedance_ohm` | `49.88827755069874` | **`49.712534512`** |
| `sample_frequency_ghz` | `5.98875` | `5.98875` (matches) |
| `s11_magnitude` | `0.0011439061772806117` | matches exactly |
| `return_loss_db` | `58.83219189478079` | matches exactly |

Only the impedance disagrees. No frequency in the sweep reproduces
`49.888278` under the current formula (nearest is `49.737204` at 11.92 GHz).
The other three quantities matching exactly proves the file and the parser
agree — the impedance number is stale.

There is **no output-file hash anywhere in the repository**, so it is
impossible to tell whether the `.s2p` changed after evidence was written or
the estimator changed. That is the provenance gap this work closes.

Two further CPW contradictions:

- Runtime: `openems_result.json` records `runtime_seconds: 1223.167`; the
  top-level README says **`1587 s`**.
- Status prose: README line 170 says CPW is **`PHYSICS_VERIFIED`**; README
  line 207 says *"**Analytical or solver-skipped:** CPW impedance and
  resonator length."* Both describe the showcase. (The *benchmark*
  `examples/benchmarks/02_cpw_50ohm/` is separately, and correctly,
  `SIMULATION INPUT PREPARED` / `ANALYTICAL ONLY` — two different artifacts
  with the same name.)

## C4 — Convergence evidence does not exist for every `PHYSICS_VERIFIED` claim

Four showcases publish `PHYSICS_VERIFIED`. Phase 4 requires that status to be
backed by convergence evidence. What actually exists:

| Showcase | Solver | Convergence control | Real evidence? |
| --- | --- | --- | --- |
| 01 IDC | FasterCap | `-a0.01` automatic refinement to 1% (in the recorded command) | yes |
| 03 IDC region | FasterCap | `-a0.01` | yes |
| 02 CPW | openEMS | energy-decay + excitation-support gate (`_openems_convergence_problem` returns `None`) | yes, for FDTD time-domain convergence; **no mesh-refinement study** |
| 04 spiral | FastHenry | **none** — the deck has no `nhinc`/`nwinc` filament discretisation and there is no refinement sweep | **no** |

A single-filament FastHenry solve cannot resolve current crowding in a spiral.
`04` therefore cannot carry `PHYSICS_VERIFIED` under the new rule and must be
restated as `SIMULATION_EXECUTED`.

## C5 — Every status is hand-maintained

No showcase has a canonical evidence record. No Markdown document marks which
of its sections are generated, so nothing can detect a stale status section and
nothing regenerates on change. This is the structural cause of C1: correcting
`openems_result.json` had no mechanism to correct the eight files derived from it.

## C6 — Stale generated counts

`PROJECT_STATUS.md` reports **`512 passed, 0 failed, 0 skipped`** for
`tests/textlayout_suite`. The suite actually reports **616 passed**. The number
is read from a stale `out/evidence/test_report.xml`, and the freshness test does
not compare it against a fresh run.

## Root cause

Status was a *string copied into each artifact at generation time*, not a value
*derived from one record at render time*. There was no content-addressed
evidence, so:

1. correcting an upstream file could not propagate downstream;
2. a solver output could change after its evidence was written and nothing
   would notice (C2);
3. a validator could pass while the artifacts contradicted each other, because
   each file was checked against itself.
